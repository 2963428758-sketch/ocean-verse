## 架构决策记录

### 决策 1：注入 ChatModel 而非 ChatClient

**问题**：项目需要两种模型——qwen-plus（问答）和 qwen-vl-max（视觉识别），如何管理？

**备选方案**：
- A. 创建两个 ChatClient bean + @Qualifier 注入
- B. 注入 ChatModel，按需构建 ChatClient（✅ 采用）

**选择 B 的原因**：
- 避免 @Qualifier + @RequiredArgsConstructor 的 Lombok 兼容性问题（Lombok 不传播字段上的 @Qualifier 到构造器参数）
- 代码更灵活：`ChatClient.create(chatModel).prompt().options(DashScopeChatOptions.builder().withModel(name).build())` 可动态切换模型
- 无需 @Primary 注解，避免未来扩展时的歧义错误

### 决策 2：DashScopeChatOptions 包路径

**发现**：Spring AI Alibaba 1.0.0.3 中，`DashScopeChatOptions` 的正确包路径是：
```
com.alibaba.cloud.ai.dashscope.chat.DashScopeChatOptions
```
而非 Spring AI 标准路径 `org.springframework.ai.dashscope`。这是通过 `jar tf` 反编译实际 jar 确认的。

### 决策 3：使用 .stream().content() 而非 .chatResponse()

**确认的 API 签名**（通过 `javap` 反编译确认）：
```java
// StreamResponseSpec 接口
Flux<String> content();                         // 直接返回文本流（✅ 采用）
Flux<ChatResponse> chatResponse();              // 返回完整响应对象流
Flux<ChatClientResponse> chatClientResponse();  // 返回客户端响应流
```

选择 `.content()` 因为它直接返回 `Flux<String>`，代码更简洁。

### 决策 4：media(MimeType, URL) 直接传图片 URL

```java
// PromptUserSpec 接口
PromptUserSpec media(MimeType, URL);       // 直接传 URL（✅ 采用）
PromptUserSpec media(MimeType, Resource);  // 传 Spring Resource
PromptUserSpec media(Media...);            // 传 Media 对象
```

选择 `(MimeType, URL)` 因为 OSS 返回的就是 URL 字符串，无需额外包装。

### 决策 5：SseEmitter 管理 SSE 连接生命周期（修复后新增）

**问题**：原方案中 `chatStream()` 在 Service 层直接操作 `HttpServletResponse.getWriter()`，调用 `.subscribe()` 后方法立即返回，Tomcat 回收 response 对象，而 Reactor 线程仍在异步写入，导致 `IllegalStateException: The response object has been recycled`。

**解决方案**：
- **Service 层**：`chatStream()` 签名改为 `Flux<String> chatStream(ChatDTO)`，只负责构建 Flux 管道（doOnNext 累积回答、doOnComplete 存库），不再 subscribe、不再接触 HTTP 响应对象
- **Controller 层**：创建 `SseEmitter(120_000L)`，订阅 Service 返回的 Flux，通过 `emitter.send(SseEmitter.event().data(...))` 推送事件。SseEmitter 告诉 Tomcat "连接未结束"，阻止 response 被回收

```java
// Controller — 生命周期由 SseEmitter 管理
@PostMapping("/chat")
public SseEmitter chat(@RequestBody ChatDTO dto) {
    SseEmitter emitter = new SseEmitter(120_000L);
    aiService.chatStream(dto).subscribe(
        chunk -> emitter.send(SseEmitter.event().data(json)),
        error -> { emitter.send(...); emitter.complete(); },
        ()    -> { emitter.send(data("[DONE]")); emitter.complete(); }
    );
    return emitter;
}
```

### 决策 6：SSE 数据 JSON 编码解决换行丢失（修复后新增）

**问题**：Spring `SseEmitter.event().data(content)` 不会转义内容中的 `\n`，而 SSE 协议将 `\n` 视为事件分隔符。当 AI 模型返回含换行的 markdown 内容时，换行被协议层吃掉，前端收到的文本全部挤在一行。

**排查过程**：
1. 首先尝试后端 `chunk.split("\n", -1)` 逐行发送 + 前端 pending 缓冲拼接 → 失败：DashScope 流式 chunk 本身是很小的文本片段（"全球"、"许多"），不含 `\n`，但每个片段被当作独立事件后，前端在每个片段后都插入了 `\n`，导致每个词都换行
2. 根因：前端无法区分"两个 SSE 事件来自同一 chunk 的 `\n` 拆分"还是"来自两个不同的 chunk"

**最终方案**：
- **后端**：`objectMapper.writeValueAsString(chunk)` 对每个 chunk 做 JSON 编码，`\n` 被转义为 `\\n`（字面两字符），SSE 数据行内无真实换行
- **前端**：`JSON.parse(content)` 解码还原原始文本，`\n` 正确恢复为换行符

```
数据流：
AI chunk "第一行\n第二行"
  → 后端 JSON 编码 → "第一行\n第二行"（带引号，\n 是字面量）
  → SSE 数据行 → data:"第一行\n第二行"（无真实换行，安全传输）
  → 前端 JSON.parse → "第一行\n第二行"（还原为含真实换行的字符串）
  → marked 渲染 → 第一行<br>第二行
```

## Bug 修复记录

### BUG-1：JwtInterceptor 拦截 AI 接口（401）

**现象**：所有 `/api/ai/*` 请求返回 `{code:401, message:'未登录或token无效'}`

**根因**：项目 API 认证由 `WebConfig` 的 `JwtInterceptor`（HandlerInterceptor）控制，`SecurityConfig` 仅 permitAll 兜底。新加的 AI 接口未添加到拦截器排除列表。

**修复**：`WebConfig.excludePathPatterns` 添加 `"/api/ai/**"`

### BUG-2：DashScope qwen-vl-max "url error"

**现象**：图像识别请求返回 `{"errorMessage":"url error"}`

**排查**：最初怀疑 OSS 的 `Content-Disposition: attachment` 导致模型无法访问图片，尝试用 `ossClient.generatePresignedUrl()` 生成签名 URL → 仍报错

**根因**（通过阿里云官方文档确认）：`qwen-vl-max` 是多模态视觉模型，必须在 `DashScopeChatOptions` 中设置 `.withMultiModel(true)`，否则请求走纯文本端点 `/generation` 而非多模态端点 `/multimodal-generation`，后者才支持图片 URL 参数。

**修复**：
```java
ChatClient.create(chatModel).prompt()
    .options(DashScopeChatOptions.builder()
        .withModel(aiProperties.getImageModel())
        .withMultiModel(true)    // ← 关键：强制走多模态端点
        .build())
    .user(u -> u.text(prompt).media(mimeType, imageUrl))
    .call().content();
```

**附注**：修复后签名 URL 不再必要，已回退 `AiServiceImpl` 中的图片 URL 为 OSS 原始地址。但 `OssUtil.generateSignedUrl()` 方法保留，供后续使用。

### BUG-3：SSE response recycled（详见决策 5）

**现象**：`IllegalStateException: The response object has been recycled and is no longer associated with this facade`

**根因**：Service 层 `.subscribe()` fire-and-forget，方法返回后 Tomcat 回收 response，Reactor 线程仍在写入

**修复**：Service 层返回 `Flux<String>`，Controller 层用 `SseEmitter` 订阅

### BUG-4：SSE 换行符丢失（详见决策 6）

**现象**：前端 AI 回答内容全部挤在一行，无换行无段落

**根因**：`SseEmitter.event().data()` 不转义 `\n`，换行被 SSE 协议当作事件分隔符

**修复**：后端 JSON 编码 + 前端 JSON.parse 解码

### BUG-5：前端 SSE 解析格式不匹配

**现象**：浏览器 EventStream 面板显示有数据，但页面无内容输出

**根因**：前端用 `line.startsWith('data: ')` 解析（冒号后有空格），但 Spring SseEmitter 生成的格式是 `data:content`（无空格），所有行匹配不上，`onChunk` 从未被调用

**修复**：兼容两种前缀格式 + 添加跨 chunk 行缓冲 + JSON 解码

### BUG-6：前端 markdown 未渲染

**现象**：AI 返回的 markdown 语法（`**加粗**`、`- 列表`）以纯文本显示

**修复**：安装 `marked` 库，assistant 消息用 `v-html="renderMarkdown(content)"` 渲染，添加 `.markdown-body` 排版样式（`:deep()` 穿透 scoped）

## 已知限制（后续优化）

1. **SSE 连接取消**：当前客户端断开后模型仍会继续生成直到完成。后续可用 `SseEmitter.onCompletion()` + `Disposable.dispose()` 实现优雅取消。

2. **历史查询无用户隔离**：`getRecognitionHistory()` 和 `getChatHistory()` 返回所有用户的数据。后续应添加 `.eq(Entity::getUserId, getCurrentUserId())` 过滤。

3. **单条 SSE 数据量**：JSON 编码解决了换行问题，但每个 chunk 都是独立事件。如果 DashScope 返回超大 chunk（> 8KB），可能触发某些代理的缓冲限制。后续可设置 chunk 大小上限。

4. **图片 MIME 类型**：当前从 `file.getContentType()` 动态获取，但如果客户端未正确设置 Content-Type（如某些旧版 Apifox），可能回退为 `image/jpeg`。

## 关键代码片段

### AiServiceImpl.chatStream() — 最终实现

```java
@Override
public Flux<String> chatStream(ChatDTO dto) {
    long startTime = System.currentTimeMillis();
    QaHistory history = new QaHistory();
    // ... 初始化 history 骨架 ...

    StringBuilder fullAnswer = new StringBuilder();
    return ChatClient.create(chatModel).prompt()
            .system(CHAT_SYSTEM_PROMPT)
            .user(dto.getQuestion())
            .stream()
            .content()
            .filter(chunk -> chunk != null && !chunk.isEmpty())
            .doOnNext(fullAnswer::append)
            .doOnComplete(() -> {
                history.setAnswerText(fullAnswer.toString());
                history.setProcessingTimeMs((int)(System.currentTimeMillis() - startTime));
                qaHistoryMapper.insert(history);
                publishChatEvent(history);
            })
            .doOnError(e -> log.error("AI 流式调用异常", e));
}
```

### AiController.chat() — SseEmitter + JSON 编码

```java
@PostMapping("/chat")
public SseEmitter chat(@RequestBody ChatDTO dto) {
    SseEmitter emitter = new SseEmitter(120_000L);
    aiService.chatStream(dto).subscribe(
        chunk -> {
            String json = objectMapper.writeValueAsString(chunk);  // \n → \\n
            emitter.send(SseEmitter.event().data(json));
        },
        error -> { emitter.send(data("[ERROR]")); emitter.complete(); },
        ()    -> { emitter.send(data("[DONE]")); emitter.complete(); }
    );
    return emitter;
}
```

### 前端 SSE 解析 — JSON 解码 + 行缓冲

```typescript
function processLine(line: string) {
  let content = extractDataField(line)   // 兼容 'data:' 和 'data: '
  if (content === '[DONE]') { finished = true; onDone(); return }
  try {
    const text = JSON.parse(content)      // 还原 \n
    if (text) onChunk(text)
  } catch {
    if (content) onChunk(content)         // 降级：非 JSON 时直接使用
  }
}
```
