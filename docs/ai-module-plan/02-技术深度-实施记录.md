## 架构决策记录

### 决策 1：SimpleVectorStore 而非 RedisVectorStore

**问题**：docker-compose 使用 `redis:7-alpine`（标准 Redis），不包含 RediSearch 模块。

**方案**：使用 Spring AI 内置的 `SimpleVectorStore`（内存向量存储），零额外依赖，`spring-ai-vector-store-1.0.0.jar` 已通过 DashScope Starter 传递依赖引入。

**后续升级路径**：如需持久化向量存储，切换到 `redis/redis-stack-server` 镜像 + `spring-ai-starter-vector-store-redis` 依赖即可。

### 决策 2：ChatClient.messages() 实现多轮对话

**验证的 API**（通过 `javap` 反编译确认）：
```java
// ChatClient.ChatClientRequestSpec
ChatClientRequestSpec messages(Message...);
ChatClientRequestSpec messages(List<Message>);  // ← 采用
```

**消息构造**：
```java
List<Message> messages = new ArrayList<>();
messages.add(new SystemMessage(systemPrompt));   // System + RAG 上下文
messages.addAll(historyMessages);                  // 历史对话 (User/Assistant 交替)
messages.add(new UserMessage(currentQuestion));    // 当前提问
```

### 决策 3：DashScope EmbeddingModel 自动配置

**确认**：`DashScopeEmbeddingAutoConfiguration`（在 `spring-ai-alibaba-autoconfigure-dashscope-1.0.0.3.jar` 中）自动创建 `EmbeddingModel` Bean，使用 `application.yml` 中配置的 `text-embedding-v3` 模型。无需手动配置。

### 决策 4：SseEmitter 订阅取消（代码质量修复）

**问题**：客户端断开后 Flux 继续运行，模型白白消耗 Token。

**修复**：
```java
Disposable subscription = aiService.chatStream(dto).subscribe(...);
emitter.onTimeout(() -> subscription.dispose());
emitter.onCompletion(() -> subscription.dispose());
```

### 决策 5：SimpleVectorStore 持久化策略

**问题**：每次应用重启都要重新调用 text-embedding-v3 向量化全部物种数据，浪费 API Token 且拖慢启动速度。

**方案**：利用 `SimpleVectorStore` 自带的 `save(File)` / `load(File)` 机制，将向量索引序列化为 JSON 文件。

**生命周期**：
- `@PostConstruct`：检查 `data/ai/vector-store.json` 是否存在 → 存在则 `load()`，否则从 DB 加载并 `save()`
- `@PreDestroy`：调用 `save()` 保存最新索引

**路径可配**：通过 `oceanverse.ai.vector-store-path` 配置，默认 `data/ai/vector-store.json`。

## Bug 修复记录（代码审查发现）

### BUG-7：ChatDTO.question 无校验（NPE 风险）

**现象**：发送 `{"question": null}` 导致 NPE 在 RAG 检索和 Spring AI 内部深处。

**修复**：`ChatDTO.question` 添加 `@NotBlank` 注解，Controller 添加 `@Valid`。

### BUG-8：SseEmitter 超时后 Flux 未取消

**现象**：120 秒超时后模型仍在生成，浪费 Token 和计算资源。

**修复**：保存 `Disposable` 引用，在 `onTimeout`/`onCompletion` 回调中调用 `dispose()`。

### BUG-9：DocumentLoader.selectList() 返回值未做空判断

**修复**：添加 `null` 和 `isEmpty()` 双重检查，空表时返回 `List.of()`。

## 优化变更记录（2026-06-17）

针对初版实施记录中列出的 5 项已知限制，完成 4 项优化（第 5 项为架构特性，保留为后续优化）：

### 优化 1：SimpleVectorStore 持久化（原限制 1）

**变更**：`KnowledgeBaseService` 新增 `saveOnShutdown()`（`@PreDestroy`）和启动时磁盘加载逻辑。

**行为**：
- 启动时：优先从 `data/ai/vector-store.json` 加载已有索引，文件不存在时才从数据库加载并保存
- 关闭时：自动保存索引到磁盘
- 效果：重启不再消耗 embedding API Token

**涉及文件**：`KnowledgeBaseService.java`、`AiProperties.java`（新增 `vectorStorePath`）

### 优化 2：RAG 相似度阈值可配置（原限制 2）

**变更**：`KnowledgeBaseService.search()` 中 `similarityThreshold` 从硬编码 `0.5` 改为 `aiProperties.getSimilarityThreshold()`。

**配置**：`application.yml` 中 `oceanverse.ai.similarity-threshold: 0.5`

### 优化 3：会话 TTL 可配置（原限制 3）

**变更**：`ChatSessionManager.saveMessage()` 中 `expire()` 从硬编码 `30` 改为 `aiProperties.getSessionTtlMinutes()`。

**配置**：`application.yml` 中 `oceanverse.ai.session-ttl-minutes: 30`

### 优化 4：maxTokens/temperature 生效（原限制 4）

**变更**：`AiServiceImpl.chatStream()` 中 `DashScopeChatOptions.builder()` 新增 `.withMaxToken()` 和 `.withTemperature()` 调用。

## 已知限制（后续优化）

1. **Redis 会话操作非原子**：saveMessage 中的 rightPush/trim/expire 不是原子操作，高并发下可能出现轻微的计数偏差。如需严格一致性，可改用 Lua 脚本。

2. **向量索引文件格式**：SimpleVectorStore 持久化为 JSON 文件，大规模数据（>10 万条）时加载较慢。后续可切换到 RedisVectorStore（需升级 Redis 镜像）。

3. **RAG 无增量更新**：知识库每次启动全量加载 species 表。后续可添加增量同步机制（监听 species 表变更事件）。
